# ADR-0007: 도메인 불변식 자가 검증(Self-Validation)을 도입하고 외부 Validator를 제거한다

## 상태

Accepted

## 날짜

2026-05-29

## 맥락

현재 상품명과 옵션명 등 비즈니스 제약 조건을 검증하기 위해 `ProductNameValidator` 및 `OptionNameValidator`와 같은 헬퍼 객체를 사용하고 있다. 이 방식은 다음과 같은 문제점을 낳는다.
1. **도메인 모델의 빈혈화(Anemic Domain Model)**: 핵심 도메인 규칙(불변식)이 엔티티가 아닌 외부 검증 클래스에 절차지향적으로 나뉘어 있어, 응집도가 떨어진다.
2. **도메인 무결성 훼손 가능성**: 실수로 서비스나 컨트롤러에서 검증 클래스를 호출하지 않은 채 엔티티를 생성하거나 수정할 경우, 유효하지 않은 불변식을 지닌 엔티티가 영속성 컨텍스트에 진입할 수 있다.

따라서 객체지향 및 DDD 관점에서 도메인 객체가 항상 유효한 상태(Invariant Protection)를 직접 유지하도록 구조를 개선해야 한다.

## 결정

1. 기존 외부 헬퍼 클래스인 `ProductNameValidator` 및 `OptionNameValidator`를 **완전히 삭제**한다.
2. `Product`와 `Option` 엔티티 생성자 및 상태 변경 메서드(`update` 등) 내부로 모든 이름 제약 조건 검증 논리를 밀어 넣는다.
3. 검증 실패 시 기존과 동일하게 도메인 전용 예외인 `CatalogException.invalid(...)`를 던진다.
4. 단위 테스트 역시 Validator 객체를 Mocking하거나 직접 호출하는 대신, `Product`와 `Option` 엔티티를 부적절한 값으로 생성하여 예외가 발생하는지 자가 검증을 수행하도록 변경한다.

## 트레이드오프

### 장점 (Pros)
- **캡슐화 및 응집도 극대화**: 엔티티 자체가 자신의 규칙을 완벽히 알며 스스로를 검증하므로, 도메인 불변식이 강력히 보호된다.
- **클래스 및 의존성 감소**: 불필요한 유틸리티성 검증 클래스가 제거되어 코드베이스가 명료해진다.
- **서비스 계층 단순화**: 애플리케이션 서비스에서 복잡하게 검증 클래스를 주입받아 호출할 필요 없이, 단순히 엔티티를 생성하거나 호출하는 것만으로 검증이 안전하게 보장된다.

### 단점 (Cons)
- **엔티티 크기 증가**: 엔티티 내부에 정규식이나 글자 수 체크 같은 유효성 검증 코드가 직접 작성되므로 파일이 다소 길어질 수 있다. (단, 이 제약조건 역시 비즈니스의 핵심 규칙이므로 엔티티가 가지는 것이 옳다.)

## 결과

- `spring-gift` 서비스는 절차지향적인 헬퍼 기반 검증을 탈피하고 진정한 DDD 풍부한 도메인 모델(Rich Domain Model)로 전환한다.
- 검증 규칙 변경이 필요할 때 오직 엔티티 내부만 수정하면 되므로 변경의 영향 범위가 축소된다.
