# ADR-0009: 일관성(Consistency)과 직관성(Intuitiveness)을 시스템 전체 설계와 검증의 제1원칙으로 세운다

## 상태

Accepted

## 날짜

2026-05-29

## 맥락

소프트웨어 시스템이 오래 지속되고 견고하기 위해 가장 중요한 가치는 **예측 가능성(Predictability)**이다. 훗날 작성자가 아닌 제3자가 코드를 유지보수할 때, 시스템의 스타일과 의도를 직관적으로 예측할 수 있어야 인지 과부하 없이 코드를 안정적으로 수정할 수 있다.

현재 코드베이스는 모듈러 모놀리스를 지향하고 있으나, 다음과 같은 사소한 불일치 지점들이 존재하여 유지보수 시 직관성을 떨어뜨린다.
1. **메인 코드와 테스트 패키지 구조의 불일치**: 메인 패키지에는 `gift.auth`와 `gift.member`가 분리되어 있으나, 테스트 패키지에는 `auth` 폴더가 없고 `member` 패키지에 인증 테스트가 섞여 있다.
2. **예외 처리 방식의 불일치**: `catalog`, `member`, `point`, `wish` 도메인은 오류 발생 시 도메인 전용 예외(`CatalogException` 등)를 던져 전역 핸들러에서 일괄 처리하지만, `order` 도메인은 일부 오류 상황(옵션 누락 등)에서 비즈니스 예외 대신 `CreateResult`라는 커스텀 결과 레코드를 리턴하고 있다.
3. **테스트 스타일의 미세한 불일치**: 각 테스트 클래스마다 Mockito 설정 방식이나 테스트 구조(Given/When/Then 주석 유무 및 검증 스타일)에 미세한 차이가 있어 읽는 흐름이 끊길 수 있다.

따라서 일관성과 직관성을 최우선 가치로 삼고, 이를 확실한 구조 규칙으로 통일해야 한다.

## 결정

### 1. 패키지 및 디렉토리 구조의 완벽한 대칭성 보장
- `src/main/java/gift` 하위 패키지 구성과 `src/test/java/gift` 하위 패키지 구성을 **100% 일치**시킨다.
- 회원 패키지(`gift.member`)와 분리되어 있는 인증 도메인(`gift.auth`)에 맞춰, 테스트 패키지에도 `gift.auth` 패키지를 신설하고 관련 테스트를 이동시킨다.

### 2. 예외 처리 및 결과 반환 모델의 통일
- `Order` 도메인에도 도메인 전용 예외인 `OrderException`을 도입한다.
- `OrderService.createOrder`에서 옵션이 없는 상황에 `CreateResult.optionMissing()`을 반환하는 대신, 타 도메인과 일관되게 `OrderException.notFound("주문 옵션을 찾을 수 없습니다. id=" + optionId)`를 발생시킨다.
- 이로써 모든 도메인 서비스는 **"성공 시 정상 응답 반환 / 실패 시 비즈니스 예외 Throw"**라는 일관된 흐름을 타며, 전역 예외 처리기(`GlobalApiExceptionHandler`)가 이를 직관적으로 처리하게 만든다.

### 3. 테스트 코드 구조의 표준화 (Given-When-Then)
- 모든 단위 테스트와 검증 코드는 예외 없이 다음의 표준화된 주석 블록 규칙을 따른다:
  ```java
  // given
  // 준비 코드 및 모킹

  // when
  // 단일 실행 흐름

  // then
  // 단일/다중 Assertion 및 Mock 검증
  ```
- `@DisplayName`은 검증하고자 하는 비즈니스 요구사항을 한글 문장형으로 통일하여 명시한다.

## 트레이드오프

### 장점 (Pros)
- **인지 부하의 획기적 감소**: 개발자가 특정 코드를 탐색할 때, 패키지 구조나 예외 전파 방식, 테스트 작성법을 100% 예측할 수 있으므로 코드 파악 속도가 매우 빨라진다.
- **아키텍처 붕괴 방지**: "이 시스템은 항상 이렇게 짜여 있다"라는 무언의 규칙이 코드 전체에 직관적으로 각인되어, 후임 개발자가 코드를 추가할 때도 자연스럽게 기존 스타일을 모방하게 된다.
- **코드의 간결함**: 성공/실패 여부를 담은 복잡한 레코드 분기 처리가 없어지고 표준 예외 흐름으로 통합되어 코드가 간결해진다.

### 단점 (Cons)
- **일부 보일러플레이트 구조화 비용**: 일관성을 맞추기 위해 예외 클래스를 새로 정의하고 테스트 코드를 재배치하는 물리적인 리팩토링 공수가 소폭 발생한다.

## 결과

- `spring-gift` 시스템은 일관성(Consistency)과 직관성(Intuitiveness)을 최고의 철학으로 갖춘 견고한 아키텍처로 탈바꿈한다.
- 훗날 어떤 유지보수자가 오더라도 시스템의 의도와 설계 철학을 단 5분 안에 이해할 수 있게 된다.
