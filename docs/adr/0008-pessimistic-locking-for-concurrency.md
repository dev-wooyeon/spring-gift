# ADR-0008: 동시성 처리를 위해 재고 예약 시 비관적 락(Pessimistic Lock)을 도입한다

## 상태

Accepted

## 날짜

2026-05-29

## 맥락

주문 생성 흐름에서 상품 옵션 재고(수량)를 차감하는 행위는 아주 빈번히 일어난다.
기존 코드에서는 `OptionService.reserveOption` 시점에 단순 조회를 수행한 뒤 수량을 차감하고 저장한다.
동시에 여러 사용자가 동일한 옵션에 대해 주문을 생성할 경우, 둘 이상의 트랜잭션이 동일한 재고 수량을 조회하여 수정하는 **갱신 유실(Lost Update) 또는 초과 판매(Overselling)** 현상이 발생하게 된다. 이는 실제 서비스 운영 시 심각한 비즈니스 장애로 이어진다.

따라서 데이터베이스 단에서 확실한 원자성과 정합성을 유지하기 위해 동시성 제어 전략을 수립해야 한다.

## 결정

1. `OptionRepository`에 JPA `@Lock(LockModeType.PESSIMISTIC_WRITE)`을 설정한 `findByIdWithLock` 조회 메서드를 정의한다.
2. 주문 예약 흐름(`OptionService.reserveOption`)에서는 일반 `findById` 대신 `findByIdWithLock`를 통해 대상 상품 옵션의 Row Lock을 획득한다.
3. 데이터베이스 수준에서 해당 Row에 락이 걸려있는 동안 다른 트랜잭션의 읽기/쓰기를 대기시킴으로써 정합성을 확실히 보장한다.

## 트레이드오프

### 장점 (Pros)
- **최상의 데이터 정합성**: 동시성 경쟁이 심해도 재고 차감 과정이 완전히 직렬화되어 초과 판매나 데이터 갱신 누락이 발생하지 않는다.
- **롤백 비용 최소화**: 낙관적 락(Optimistic Lock)처럼 충돌 발생 시 비즈니스 재시도 로직을 정교하게 구현하거나 수많은 사용자 요청이 실패해 롤백되는 비용을 방지하고, 단순히 락 획득 대기를 유도한다.

### 단점 (Cons)
- **성능 제약 (대기 시간)**: 동일한 상품 옵션에 주문이 한꺼번에 몰릴 때 데이터베이스 커넥션 대기가 늘어날 수 있다.
- **데드락 위험**: 복수의 테이블이나 다중 Row에 대해 상호 락을 획득하는 복잡한 트랜잭션 구조가 추가될 경우 데드락(Deadlock)이 발생할 수 있다. (현재 주문 흐름에서는 단일 옵션 Row에 대해서만 락을 얻으므로 데드락 가능성은 희박하다.)

## 결과

- `spring-gift` 시스템은 실질적인 대규모 동시 요청 운영 상황에서도 재고 정합성을 안전하게 지켜낼 수 있게 된다.
- 주문 및 재고 관리는 단순 데이터 보존을 넘어 비즈니스 신뢰성의 핵심이므로 성능 소폭 감소(Lock 획득 비용)보다 데이터 무결성을 절대 우선시한다.
