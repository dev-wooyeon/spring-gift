# ADR-0003: 대규모 리팩토링 전 JaCoCo 커버리지 게이트를 우선 도입한다

## 상태

Accepted

## 날짜

2026-05-13

## 맥락

프로젝트의 다음 큰 방향은 안티패턴을 제거하고, 클린 코드와 SOLID 원칙을 지키는 구조로 대규모 리팩토링하는 것이다. 이 과정에서 기존 작동은 반드시 유지되어야 한다.

현재 `./gradlew check`는 Testcontainers 기반 통합 테스트를 포함하는 기본 품질 게이트로 정해졌다. 다음으로 `check`에 붙일 검증 후보는 JaCoCo 커버리지 기준과 ArchUnit 아키텍처 규칙이다.

## 결정

다음 검증은 JaCoCo 커버리지 게이트로 정한다.

대규모 리팩토링 전에는 구조 규칙을 먼저 강제하기보다, 기존 동작을 충분한 테스트로 고정하는 일이 우선이다. 따라서 도메인 종류와 관계없이 기존 동작을 관찰 가능한 테스트로 보강하고, 그 테스트가 실제 코드 경로를 실행하는지 JaCoCo로 확인한다.

ArchUnit은 리팩토링으로 목표 패키지 구조와 의존 방향이 명확해진 뒤 도입한다.

## 원칙

- 커버리지는 리팩토링 안전망이다. 숫자 자체가 목적이 아니라 기존 동작을 깨뜨리지 않기 위한 최소 장치다.
- 테스트는 단순히 라인을 지나가게 만드는 방식이 아니라 HTTP 응답, DB 상태, 예외, 도메인 규칙처럼 관찰 가능한 동작을 검증해야 한다.
- 커버리지 기준을 통과시키기 위해 테스트를 비활성화하거나, 의미 없는 assertion을 추가하거나, 운영 코드를 임의로 제외하지 않는다.
- 커버리지 제외는 생성 코드, 애플리케이션 부트스트랩처럼 검증 가치가 낮은 코드에 한정하고, 필요하면 문서로 남긴다.
- 리팩토링 대상 코드는 먼저 characterization test로 현재 동작을 고정한 뒤 변경한다.

## 트레이드오프

### JaCoCo

장점:

- 리팩토링 전에 테스트가 실제 코드 경로를 충분히 실행하는지 확인할 수 있다.
- `./gradlew check`에 연결해 로컬과 CI에서 같은 기준을 강제할 수 있다.
- 테스트 보강이 필요한 영역을 수치와 리포트로 확인할 수 있다.
- 기존 동작 보존을 목표로 하는 characterization test와 잘 맞는다.

단점:

- 커버리지 숫자는 동작 보존을 완전히 보장하지 않는다.
- 기준값을 너무 빨리 높이면 의미 있는 테스트 작성보다 숫자 맞추기가 먼저 될 수 있다.
- 통합 테스트 중심이면 실행 시간이 늘어날 수 있다.

### ArchUnit

장점:

- 리팩토링 이후 목표 아키텍처를 코드 규칙으로 강제할 수 있다.
- Controller, Repository, domain 간 의존 방향을 지속적으로 보호할 수 있다.

단점:

- 현재는 리팩토링 목표 구조가 아직 코드로 정착되지 않았다.
- 구조 규칙을 먼저 강제하면 기존 동작 보존보다 패키지 모양 맞추기가 먼저 될 수 있다.

## 결과

- 다음 구현 작업은 JaCoCo를 `./gradlew check` 품질 게이트에 연결하는 것이다.
- JaCoCo 도입 시 현재 커버리지 기준선을 먼저 확인하고, 대규모 리팩토링 전에 필요한 characterization test를 보강한다.
- ArchUnit은 리팩토링 후 목표 구조가 명확해진 시점에 별도 ADR 또는 Todo로 다시 결정한다.
