# Admin 페이지 확인 및 비즈니스 분석

## 분석 원칙

- 제품 사용자 관점에서 먼저 분석한다.
- 화면과 API를 역엔지니어링하여 비즈니스 흐름을 파악한다.
- 비즈니스 흐름을 정리한 뒤 컨트롤러, 템플릿, 도메인 코드를 분석한다.

## 현재 접근 제어 기준선

2026-05-10 기준, 프로젝트에는 Spring Security 설정이 없다.

- `/admin/products`: 인증 없이 `200 OK`
- `/admin/products/new`: 인증 없이 `200 OK`
- `/admin/products/1/edit`: 인증 없이 `200 OK`
- `/admin/members`: 인증 없이 `200 OK`
- `/admin/members/new`: 인증 없이 `200 OK`
- `/admin/members/1/edit`: 인증 없이 `200 OK`
- `/api/products`: 인증 없이 `200 OK`
- `/api/wishes`: 인증 헤더가 없으면 `400 Bad Request`

확인 명령:

```bash
./gradlew bootRun --args='--server.port=18080'
curl -i -s http://localhost:18080/admin/products
curl -i -s http://localhost:18080/admin/members
curl -i -s http://localhost:18080/api/products

```

주의: 기본 `8080` 포트가 사용 중이면 위처럼 임시 포트로 실행해 확인한다.

## Admin 페이지 확인 단계

### 1단계: 기본 접근 확인

- 브라우저 또는 `curl`로 `/admin/products`, `/admin/members`를 직접 호출한다.
- 로그인, 접근 키, Authorization 헤더 없이 페이지가 열리면 현재 상태를 "관리자 화면 무인증 접근 가능"으로 기록한다.
- 이후 보안 설정이 추가되면 같은 요청이 로그인 화면 이동, `401`, `403` 중 하나로 바뀌어야 한다.

### 2단계: 상품 관리 화면 확인

상품 관리 업무는 `AdminProductController`와 `product/*.html` 템플릿이 담당한다.

- 목록: `/admin/products`
- 추가 화면: `/admin/products/new`
- 수정 화면: `/admin/products/{id}/edit`
- 생성 POST: `POST /admin/products`
- 수정 POST: `POST /admin/products/{id}/edit`
- 삭제 POST: `POST /admin/products/{id}/delete`

확인할 조건:

- 상품명은 필수이며 최대 15자다.
- 상품명 허용 문자는 한글, 영문, 숫자, 공백, `( ) [ ] + - & / _`다.
- 관리자 상품 화면에서는 `카카오`가 포함된 상품명을 허용한다.
- 가격은 화면에서 `min=1`, API 요청 모델에서 `@Positive`로 표현된다.
- 이미지 URL과 카테고리는 필수다.
- Spring Security를 나중에 추가하면 모든 POST form에 CSRF 토큰 적용 여부를 확인해야 한다.

### 3단계: 회원 관리 화면 확인

회원 관리 업무는 `AdminMemberController`와 `member/*.html` 템플릿이 담당한다.

- 목록: `/admin/members`
- 추가 화면: `/admin/members/new`
- 수정 화면: `/admin/members/{id}/edit`
- 생성 POST: `POST /admin/members`
- 수정 POST: `POST /admin/members/{id}/edit`
- 포인트 충전 POST: `POST /admin/members/{id}/charge-point`
- 삭제 POST: `POST /admin/members/{id}/delete`

확인할 조건:

- 회원 추가 시 이메일 중복이면 오류 메시지를 보여준다.
- 포인트 충전 금액은 화면에서 `min=1`, 도메인에서 0 이하를 거부한다.
- 회원 수정 화면은 현재 비밀번호 값을 `value`에 렌더링한다. 이는 보안 리스크로 기록한다.
- 회원 목록에서는 이메일과 포인트가 노출된다.

### 4단계: 회귀 기준 기록

- 관리자 HTML 확인 중 `/api/**` 동작은 변경하지 않는다.
- `/api/products`는 현재 인증 없이 상품 목록을 반환한다.
- `/api/members/register`, `/api/members/login`은 토큰 발급 진입점이다.
- `/api/wishes`, `/api/orders`는 Authorization 헤더를 전제로 한다.

## 비즈니스 역분석

### 사용자 역할

- 고객: 상품을 조회하고, 회원가입/로그인 후 위시와 주문을 수행한다.
- 회원: 포인트를 보유하고 주문 시 포인트가 차감된다.
- 관리자: 상품, 회원, 포인트를 운영하기 위해 HTML 관리자 화면을 사용한다.

### 업무 단위

- 상품 업무: 카테고리 선택, 상품 등록, 상품 수정, 상품 삭제, 목록 확인.
- 회원 업무: 회원 등록, 회원 정보 수정, 포인트 충전, 회원 삭제, 목록 확인.
- 고객 업무: 회원가입, 로그인, 상품 조회, 위시 등록/삭제, 주문.
- 인증 업무: 일반 API는 JWT를 사용하고, 관리자 HTML은 아직 인증 정책이 없다.

### 화면과 API 연결

- 관리자 상품 화면: `AdminProductController` -> `product/list`, `product/new`, `product/edit`
- 관리자 회원 화면: `AdminMemberController` -> `member/list`, `member/new`, `member/edit`
- 고객 상품 API: `ProductController` -> `/api/products`
- 카테고리 API: `CategoryController` -> `/api/categories`
- 회원 API: `MemberController` -> `/api/members`
- 위시 API: `WishController` -> `/api/wishes`
- 주문 API: `OrderController` -> `/api/orders`

## 리스크와 다음 작업 후보

- 관리자 화면이 인증 없이 접근 가능하다.
- 관리자 회원 수정 화면이 비밀번호 값을 HTML에 포함한다.
- 상품/카테고리 생성, 수정, 삭제 API도 인증 없이 열려 있는지 별도 정책 결정이 필요하다.
- Spring Security 도입 시 `/admin/**` 보호와 `/api/**` 기존 JWT 흐름을 분리해야 한다.
- Spring Security 도입 시 관리자 POST form에 CSRF 토큰을 추가해야 한다.
