# Agent Guide

이 문서는 AI 에이전트와 개발자가 항상 지켜야 하는 작업 규칙이다. 목표는 단순 구현이 아니라, 누구나 개인 로컬 설정에 의존하지 않고 동일하게 실행할 수 있는 반복 가능한 테스트 시스템을 만드는 것이다.

## 핵심 목표

- 개인 PC에 설치된 DB, IDE 설정, 수동 환경값에 의존하지 않는다.
- `./gradlew test` 또는 명확한 단일 명령으로 검증할 수 있게 만든다.
- 프로덕션과 유사한 환경은 실제 운영과 같은 모든 인프라를 복제한다는 뜻이 아니라, 운영에서 쓸 가능성이 높은 구성요소를 자동화된 테스트 안에서 재현한다는 뜻이다.
- 이 프로젝트에서는 H2 중심 검증보다 MySQL, Flyway, Spring Boot 실행 흐름을 자동 검증하는 방향을 우선한다.
- SonarQube 같은 별도 품질 플랫폼보다 먼저, 저장소 안에서 실행 가능한 Gradle 기반 테스트와 품질 게이트를 만든다.

## 항상 지킬 프로그래밍 요구 사항

- 코드 수정을 시작하기 전에 `README.md` 또는 계획 문서에 다음 작업 한 가지가 명확히 정의되어 있어야 한다.
- 한 번에 한 조각만 변경한다.
- 구조 변경과 작동 변경을 같은 커밋에 섞지 않는다.
- 가능하면 Red, Green, Refactor 순서로 진행한다.
- 최소 기준은 변경 후 전체 테스트 통과다.
- 테스트를 회피하거나 비활성화하는 변경은 금지한다.
- AI 산출물을 그대로 신뢰하지 않고, 중간 결과와 diff를 직접 확인한다.
- 작동 변경은 상태 재조회, HTTP 응답, DB 조회 등 관찰 가능한 증거로 검증한다.
- 커밋은 하나의 목적만 가져야 하며, `git diff`를 보고 30초 안에 의도를 설명할 수 있어야 한다.
- 선택지가 2개 이상이고 트레이드오프가 있거나, 반복적으로 따를 규칙을 정하면 ADR을 작성한다.

## 작업 순서

1. 현재 작업의 목적을 한 문장으로 정리한다.
2. 기능 목록 또는 체크리스트에 다음 작업 한 가지를 추가한다.
3. 기존 동작을 확인하는 테스트 또는 기준선을 먼저 만든다.
4. 실패하는 테스트를 확인한다.
5. 가장 작은 변경으로 테스트를 통과시킨다.
6. 전체 테스트를 실행한다.
7. diff를 확인하고, 의도하지 않은 변경을 제거한다.
8. 검증 결과와 다음 작업을 문서에 남긴다.

## 테스트 시스템 방향

- 첫 단계는 Testcontainers 기반 통합 테스트다.
- Docker Desktop 또는 Docker Engine만 있으면 테스트가 실행되도록 구성한다.
- 테스트 코드가 MySQL 컨테이너를 자동으로 띄우고, Spring Boot가 해당 DB에 연결되도록 한다.
- Flyway migration으로 스키마와 초기 데이터를 구성한다.
- 외부 API는 실제 호출하지 않고 mock, fake, stub으로 대체한다.
- `./gradlew test`는 로컬과 CI에서 동일한 의미를 가져야 한다.
- 필요하면 이후 `./gradlew check` 또는 `./gradlew qualityGate`를 단일 검증 진입점으로 확장한다.

## Docker와 운영 유사 환경 기준

- Docker Compose는 사람이 직접 실행해 앱과 DB를 함께 확인하는 수동 통합 환경에 사용한다.
- Testcontainers는 자동 테스트에서 필요한 인프라를 코드로 선언하는 데 사용한다.
- 운영 유사 환경의 우선순위는 다음과 같다.
  - 실제 MySQL 버전 또는 운영에 가까운 MySQL 버전 사용
  - Flyway migration으로 DB 구성
  - 환경변수 기반 설정 주입
  - Spring Boot jar 또는 실제 애플리케이션 컨텍스트 기준 검증
  - 외부 네트워크 의존 제거

## 보안과 설정 규칙

- 어드민 키, 액세스 토큰, 클라이언트 시크릿은 GitHub 저장소나 클라이언트 코드에 포함하지 않는다.
- `application.properties`에는 안전한 기본값 또는 환경변수 참조만 둔다.
- Kakao API 같은 외부 연동은 테스트에서 실제 키와 실제 네트워크 호출에 의존하지 않는다.
- 관리자 화면 보호를 구현할 때는 `/admin/**`와 `/api/**` 인증 방식을 분리해서 다룬다.
- Spring Security를 도입하면 관리자 POST form의 CSRF 토큰 적용 여부를 반드시 검증한다.

## 현재 프로젝트 기준선

- 관리자 HTML 화면은 `AdminProductController`, `AdminMemberController`가 담당한다.
- 현재 `/admin/products`, `/admin/members`는 인증 없이 접근 가능한 상태다.
- 고객/외부 API는 `/api/products`, `/api/categories`, `/api/members`, `/api/wishes`, `/api/orders` 흐름으로 분리되어 있다.
- `/api/wishes`, `/api/orders`는 JWT Authorization 헤더를 전제로 한다.
- 자세한 현재 접근 제어 기준선과 비즈니스 분석은 `docs/분석.md`를 따른다.

## 에이전트 응답 규칙

- 요청받지 않은 대규모 리팩토링을 하지 않는다.
- 새로운 라이브러리나 아키텍처를 도입하기 전, 현재 목표와 작업 단계에 맞는지 설명한다.
- 불확실한 요구는 먼저 문서와 코드에서 확인한다.
- 구현을 진행하면 파일 변경, 테스트 결과, 남은 리스크를 짧게 보고한다.
- 테스트를 실행하지 못했으면 이유를 명확히 남긴다.
