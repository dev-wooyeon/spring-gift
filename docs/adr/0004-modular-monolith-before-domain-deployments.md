# ADR-0004: 도메인 단위 배포를 고려해 모듈러 모놀리스 경계를 먼저 만든다

## 상태

Accepted

## 날짜

2026-05-17

## 맥락

서비스가 커지면 상품, 회원/인증, 위시, 주문, 옵션 같은 도메인이 서로 다른 변경 속도와 장애 범위를 가질 수 있다. 장기적으로는 도메인 단위 배포가 가능해야 한다.

현재 코드는 패키지가 도메인별로 나뉘어 있지만 하나의 Spring Boot 애플리케이션, 하나의 DB 스키마, 같은 트랜잭션 경계 안에서 동작한다. 이 상태에서 바로 멀티서비스나 멀티모듈 배포로 쪼개면 다음 결정이 비어 있다.

- 각 도메인의 데이터 소유권
- 도메인 간 동기 호출과 이벤트 경계
- 주문, 포인트, 재고의 일관성 정책
- API contract와 내부 모델의 분리
- 배포 단위별 테스트/마이그레이션 전략

## 결정

바로 독립 배포 서비스로 분리하지 않는다. 먼저 모듈러 모놀리스 구조로 도메인 경계를 강제한다.

1. 도메인별로 `presentation`, `application`, `domain`, `infrastructure` 책임을 나눈다.
2. Controller는 해당 도메인의 application service만 호출한다.
3. Repository와 JPA Entity는 infrastructure 또는 domain 내부로 숨긴다.
4. 다른 도메인의 Entity/Repository 직접 참조를 금지하고, 필요하면 application port 또는 domain event로 연결한다.
5. ArchUnit으로 의존 방향을 `./gradlew check`에 추가한다.
6. 데이터 소유권과 트랜잭션 정책이 명확한 도메인부터 멀티모듈, 이후 독립 배포 후보로 승격한다.

## 도메인 경계 후보

- Catalog: 상품, 카테고리, 옵션
- Member/Auth: 회원, JWT, Kakao login
- Wish: 위시 목록
- Order: 주문, 포인트 차감, 재고 차감
- Notification: Kakao 메시지 발송

## 트레이드오프

### 모듈러 모놀리스 우선

장점:

- 배포와 운영 복잡도를 늘리지 않고 코드 경계를 먼저 검증할 수 있다.
- 기존 Testcontainers 기반 characterization test를 유지하면서 리팩토링할 수 있다.
- ArchUnit으로 의존 규칙을 자동화하기 좋다.
- 도메인별 독립 배포가 필요한지 실제 변경 빈도와 결합도를 보고 판단할 수 있다.

단점:

- 런타임 장애 격리는 아직 하나의 애플리케이션 수준에 머문다.
- DB가 공유되어 있으면 도메인 간 데이터 결합을 계속 감시해야 한다.

### 바로 독립 서비스 분리

장점:

- 배포와 장애 격리를 도메인별로 가져갈 수 있다.
- 팀이 커졌을 때 소유권을 명확히 하기 쉽다.

단점:

- 현재 규모에서는 네트워크, 관측성, 데이터 정합성, 배포 파이프라인 비용이 먼저 커진다.
- 주문/포인트/재고처럼 원자성 정책이 불명확한 흐름을 분산 트랜잭션 문제로 확대할 수 있다.
- 현재 테스트 기반만으로는 서비스 간 contract 회귀를 충분히 막기 어렵다.

## 결과

- 다음 구조 리팩토링은 독립 서비스 분리가 아니라 모듈러 모놀리스 경계 정리다.
- ArchUnit을 도입해 Controller -> Application Service -> Domain/Repository 의존 방향을 강제한다.
- 주문 도메인은 재고와 포인트 일관성 정책을 먼저 결정한 뒤 배포 경계를 논의한다.
- Kakao 메시지 발송은 Order 내부 구현에서 Notification 후보로 분리한다.
